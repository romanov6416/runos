

Настройки приложения "access-control" в конфигурациионном файле:
    1. Все разрешенные пользователи указываются в "users-permissions" (указывать надо mac-адрес пользователя).
    2. Если у пользователя нет дополнительных настроек, то оставляем незаполненным его словарь настроек.
    3. В дополнительных настройках доступа пользователей указываются mac-адреса тех узлов,
    на доступ к котором данный пользователь имеет права.
    4. Далее определяется на каком уровне предоставляется доступ к хосту (доступные протоколы ARP, ICMP, TCP, UDP)
    5. Для TCP и UDP портов указывать сам протокол не нужно. Требуется указать в "tcp-ports" и в "udp-ports" список
    тех портов, к которым пользователь имеет доступ.
    6. В "default-permissions" определяется к каким узлам будут иметь доступы все пользователи,
    чьи mac-адреса указаны в "user-permissions". Уровень доступ определяется аналогично как в "user-permissions".
    7. В "default-idle-timeout" можно указать idle timeout для правил по умолчанию.










Проверочные сценарии:

1. Проверка ping
    - запустить топологию с помощью скрипта launch_topo.sh

    - с помощью следующих команд изменить MAC адреса хостов h1 и h2
            "h1 ifconfig h1-eth0 hw ether 00:00:00:00:00:01"
            "h2 ifconfig h2-eth0 hw ether 00:00:00:00:00:02"

    - запустить контроллер с конфигурационным файлом network-settings.json

    - в запущенном mininet применить команду
            "h1 ping h2"

    - в результате мы увидим, что пинг между хостами h1 и h2 проходит,
      о чем свидетельствуют следующие строки в консоли mininet:
            "64 bytes from 10.0.0.2: icmp_seq=1 ttl=64 time=0.185 ms
             64 bytes from 10.0.0.2: icmp_seq=2 ttl=64 time=0.035 ms
             64 bytes from 10.0.0.2: icmp_seq=3 ttl=64 time=0.036 ms
             ..."

    - подождем пока правила пропадут с коммутаторов (по умолчанию 10 секунд).

    - в запущенном mininet применить команду
            "h2 ping h1"

    - можно увидеть, что команда ping будто зависла ...

    - команда ping будет блокироваться, о чем можно прочитать в логах Runos:
            "new PacketIn (cookie 0x8000000000000100)"
            "has not access (send ICMP error)"
            ...

    - в блокирование ping также можно убедиться с помощью tcpdump на соответствующем порту
            "IP 10.0.0.2 > 10.0.0.1: ICMP host 10.0.0.1 unreachable - admin prohibited, length 36"





2. Проверка ssh
    - запустить топологию с помощью скрипта launch_topo.sh

    - с помощью следующих команд изменить MAC адреса хостов h1 и h2
            "h1 ifconfig h1-eth0 hw ether 00:00:00:00:00:01"
            "h2 ifconfig h2-eth0 hw ether 00:00:00:00:00:02"

    - запустить контроллер с конфигурационным файлом network-settings.json

    - в запущенном mininet применить команду
            "h1 ssh h2"

    - в результате мы увидим, что ssh было мгновенно разорвано,
      о чем свидетельствует строка в mininet консоли:
            "ssh: connect to host 10.0.0.2 port 22: Connection refused"

    - также можно посмотреть вывод нашего приложения, в котором будут следующие строки:
            "new PacketIn (cookie 0x80000000000004c1)"
            "has access (new cookie 0x80000000000004c1)"
            "add cookie 0x80000000000004c1"
            ...

    - подождем пока правила пропадут с коммутаторов (по умолчанию 10 секунд).

    - в запущенном mininet применить команду
            "h2 ssh h1"

    - можно увидеть, что ssh соединение будто зависло ...

    - ssh будет блокироваться, о чем можно прочитать в логах Runos:
            "new PacketIn (cookie 0x8000000000000500)"
            "has not access (send ICMP error)"
            ...

    - в блокирование ssh также можно убедиться с помощью tcpdump на соответствующем порту
              "IP 10.0.0.2.34552 > 10.0.0.1.ssh: Flags [S], seq 2384779018, win 29200, options [mss 1460,sackOK,TS val 3838974 ecr 0,nop,wscale 9], length 0"
              "IP 10.0.0.2 > 10.0.0.1: ICMP host 10.0.0.1 unreachable - admin prohibited, length 36"





3. Проверка telnet
    - аналогично с использованием этого же конфигурационного файла можно убедиться
      в разрешении соединения h2 telnet h1
      и в блокировании соединения h1 telnet h2
